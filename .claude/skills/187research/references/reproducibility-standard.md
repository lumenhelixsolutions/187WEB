# Reproducibility Standard

Every 187RESEARCH lab artifact must be reproducible by a future researcher or agent with access to the same inputs. Apply this standard to notebooks, demos, datasets, visualizations, and computational experiments.

## FAIR data

| Principle | Requirement |
|-----------|-------------|
| **Findable** | Assign a persistent identifier (DOI, arXiv ID, Zenodo record) and descriptive metadata. |
| **Accessible** | Provide a public access URL or documented access protocol; include license. |
| **Interoperable** | Use open formats (CSV, JSON, Parquet, Markdown) and standard vocabularies. |
| **Reusable** | Include a clear license, provenance, and usage documentation. |

## Environment specification

Document and pin:

- Operating system and version.
- Programming language and exact version.
- Dependency manager and lockfile (e.g., `requirements.txt`, `poetry.lock`, `package-lock.json`, `Cargo.lock`).
- Hardware requirements if relevant (GPU, memory, CPU cores).
- Container or virtual environment definition (Dockerfile, `environment.yml`, `.venv` instructions).

## Randomness and seeds

- Set and report all random seeds (`numpy`, `random`, `torch`, TensorFlow, etc.).
- For stochastic algorithms, report multiple runs with confidence intervals when possible.
- Note any non-determinism from hardware or parallel execution.

## Provenance

- Record the source of every raw data file.
- Record every transformation step (script names, commit hashes, parameters).
- Prefer automated provenance tools (e.g., `dvc`, `git`, `snakemake`, `nextflow`, `provenance logs`).

## Version pinning

- Pin all direct and transitive dependencies.
- Record external database/API versions or data dumps used.
- Record exact download URLs and checksums for large inputs.

## Sharing artifacts

- Host code in a public or documented repository.
- Host data in a persistent repository (Zenodo, Figshare, Dryad, Hugging Face Datasets).
- Provide a README with run instructions, expected outputs, and estimated runtime.
- Include a sample output and a validation command.

## Validation checklist

- [ ] Clean environment install succeeds from lockfile.
- [ ] A fresh run produces the same primary outputs (within documented tolerance).
- [ ] All figures and tables are generated by scripts, not hand-edited.
- [ ] All external data downloads are documented with URLs and checksums.
- [ ] Every claim is labeled with an evidence-ladder rung and source route.
- [ ] A second reviewer/agent can locate every cited source.
- [ ] Secrets and API keys are excluded from the repository.

## Reproducibility statement template

> This lab was run on `{OS}` with `{language} {version}`. Dependencies are pinned in `{lockfile}`. Random seed `{seed}` was used. Raw data were retrieved from `{URL}` on `{date}` with checksum `{hash}`. All transformations are in `{script/notebook}` at commit `{commit}`. Outputs were validated by `{validation command}` and match the sample outputs in `{path}`.
