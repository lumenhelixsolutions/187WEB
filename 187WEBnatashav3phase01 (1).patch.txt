Phase 0 (recon): baseline audit against current main, terminology inventory,
file ownership map, dependency DAG, and two ADRs recording the naming
boundary (NATASHA replaces Charlotte as the generic operator; agent-charlotte
keeps its legacy name as the one allowlisted exception) and phase sequencing
(why the rest of the handoff's migration map is deferred, not done in one
pass).

Phase 1 (identity + foundation):
- 187web-ecosystem rewritten as v3 NATASHA integration index; the four
  planned-but-unbuilt v3 modules (COMPRESS/TENSION/SPARK/SCOUT) are listed
  under an explicit "not yet implemented" heading, not linked as if real.
- agent-charlotte and neuro-toxin converted to deprecation-noticed stubs —
  fully active, original directives untouched, banner naming their planned
  (unbuilt) successors.
- widow-weaver/swarm-mind/silk-sandbox: Suite-header version bump only;
  silk-sandbox's one generic "Charlotte stack" line fixed alongside it.
- AGENTS.md: three narrow, downstream-accurate edits (skill table row,
  modules heading, one runtime-status bullet) — not a full sync.
- docs/migrations/CHARLOTTE-TO-NATASHA.md: the naming rule, a file-by-file
  disposition table, and the skill-rename map.
- scripts/check-reserved-names.mjs + npm run reserved-names:check + CI step:
  enforces the boundary going forward via a standalone-word match that
  correctly ignores the agent-charlotte identifier and the charlotte_crawl
  wire flag. Also wires npm test into CI, which previously ran nowhere.
- vitest foundation tests (scripts/__tests__), vitest.config.ts widened to
  pick them up.
- Model adapters regenerated (58 skills) and the hand-maintained .grok
  mirror synced to match.

Also corrects the Phase 0 audit's own undercount: individually checking the
19-file `.claude/skills/**` list (not just the aggregate count) surfaced 11
more files using "Charlotte" as a generic label outside this pass's ownership
map (187repo-eval, 187repo, 187vibe, 187webdesign, and the full
187web-manifest reference set, including MANIFEST.xml's mascot tag). None
fixed here — several are real content rewrites, not mechanical renames — all
catalogued in the audit addendum and allowlisted under a clearly temporary
DEFERRED_SKILL_IDS bucket so the new CI gate is accurate rather than quietly
broken. Recommended as the first item of a future session.

Verified this pass: lint, typecheck, full production build, 59/59 vitest
tests, skills:validate (58 skills), adapters:drift, docs:drift,
showcase:sync, and the new reserved-names:check all pass. release:validate
fails with a pre-existing, unrelated gap (README.md, from the prior
brand-kit commit, confirmed already broken on origin/main) — documented in
the audit, deliberately not touched.

Phases 2-7 (COMPRESS/TENSION/SPARK builds, SCOUT, 187QUANTUM/187CHAIN
runtime adapters, CLI/config routing cleanup, release sync) are not
attempted — sequencing and rationale recorded in
docs/plans/natasha-v3-master-sprint.md.

+# NATASHA-ADR-001 — Naming and identity boundary
+
+**Status:** Accepted · **Date:** 2026-07-12 · **Phase:** 0/1
+
+## Context
+
+`187web-ecosystem` v2 is organized around **Charlotte**, a stylized systems
+operator personifying the ecosystem's THREAD/TUNE/CORD/CHAR/LAB modules. The
+NATASHA v3 handoff reserves the name **Charlotte for a separate technology**
+outside this repository and requires migrating the ecosystem's generic
+operator/mascot/module-label usage to **NATASHA** terminology.
+
+## Decision
+
+1. **NATASHA** is the 187WEB v3 orchestration identity and operator name,
+   replacing Charlotte in that role everywhere it appears as a *generic*
+   label (ecosystem index, agent framing prose, AGENTS.md skill table).
+2. **`agent-charlotte`** (the crawler skill) is the one place "Charlotte"
+   named a *specific capability*, not just the generic mascot. Per the
+   handoff's own carve-out ("preserve Charlotte references only in...
+   deprecated compatibility stubs... for the separate Charlotte
+   technology"), this skill is not deleted or silently renamed in place —
+   it becomes a **deprecation stub**: it keeps working under its legacy
+   name, gains a banner pointing at the planned NATASHA v3 successor
+   (`natasha-scout` / SCOUT), and is the canonical allowlisted location
+   where "Charlotte" may still appear in prose.
+3. **`neuro-toxin`** gets the same stub-with-banner treatment, pointing at
+   the planned successor `neuro-tension` / TENSION. It doesn't mention
+   Charlotte and isn't part of the naming boundary — it's included here
+   only because Phase 1 (M1.3) groups both conversions together.
+4. **Reserved-name enforcement.** A CI-gated script
+   (`scripts/check-reserved-names.mjs`) flags any *new* generic "Charlotte"
+   usage outside an explicit allowlist, so the boundary holds going
+   forward instead of relying on manual review.
+
+## Sequencing conflict found and resolved
+
+The handoff's Phase 1 (M1.3) asks for "deprecation routers" pointing
+`agent-charlotte` → `natasha-scout` and `neuro-toxin` → `neuro-tension`,
+but both target skills are scoped to **later** phases (SCOUT is Phase 3
+M3.3; TENSION is Phase 2 M2.3) — they don't exist yet at Phase 1. Rather
+than fabricate links to nonexistent skills or silently skip M1.3, the stubs
+here state plainly that the successor is **planned, not yet implemented**,
+with a pointer to this ADR and the migration doc. No dangling links; no
+overclaimed capability.
+
+## Consequences
+
+- `agent-charlotte`'s actual crawler directive is **preserved and still
+  active** under its legacy name until `natasha-scout` ships — converting
+  it to a stub that also deleted the working capability would be a
+  regression, not a migration.
+- The full `natasha-scout` / `neuro-tension` builds, plus the CLI/config
+  surfaces that still route `char`/`ch` (`config/187-aliases.json`,
+  `config/187-command-reference.json`, `docs/187-CHAR.md`,
+  `docs/187-NAMES.md`) are **not** touched in this pass — see

+# NATASHA-ADR-002 — Module boundaries and phase sequencing
+
+**Status:** Accepted · **Date:** 2026-07-12 · **Phase:** 0/1
+
+## Context
+
+The handoff's migration map lists module renames and additions as one flat
+table, but its own **Phase plan** sequences them across Phases 1–6 with
+explicit dependencies (Phase 3 depends on Phases 1/2; Phase 6 depends on
+Phase 3). Executing the flat table literally in one pass would create
+forward references to skills, routes, and config that don't exist yet.
+
+## Decision — what's in scope now vs. deferred, and why
+
+| Item | Migration map says | This pass | Why |
+|---|---|---|---|
+| `widow-weaver` (THREAD) | Preserve and upgrade | Suite-header version bump only (v2→v3) | Full upgrade (M2.1: TLDR_Distill etc., Verification Record) is Phase 2 |
+| `swarm-mind` (CORD) | Preserve and expand | Suite-header version bump only | Full upgrade (M3.1: bounded packets, FUSE) is Phase 3 |
+| `silk-sandbox` (LAB) | Preserve and strengthen | Suite-header version bump only | Full upgrade (M3.4: lab:* profiles) is Phase 3 |
+| `agent-charlotte` → `natasha-scout` | New canonical skill; retain deprecated stub | Stub converted now (M1.3); `natasha-scout` **not created** | `natasha-scout` creation is explicitly M3.3 |
+| `neuro-toxin` → `neuro-tension` | New canonical skill; retain deprecated stub | Stub converted now (M1.3); `neuro-tension` **not created** | `neuro-tension` creation is explicitly M2.3 |
+| `token-web` (COMPRESS), `idea-spark` (SPARK) | Create | **Not created** | M2.2/M2.4 — Phase 2 |
+| `187quantum`, `187chain` | Create first-class skill + runtime adapters | **Not created** | Phase 4/5. The `SKILL.md`/`references`/`templates` layer is authorable without external tools; the `tools/qchain/**` adapters wrap Qiskit/Slither/Foundry/Echidna and their `pytest` suites are meant to *verify real tool output* — neither is available in this environment, and writing them unverified would fabricate tested capability. Flagged for a session with those tools installed. |
+| `CHAR`/`char` in `config/187-aliases.json`, `config/187-command-reference.json` | Remove from active routing and autocomplete | **Not touched** | This is explicitly M6.3 (CLI parity), which the DAG places *after* Phase 3 creates `natasha-scout`. +| `docs/187-CHAR.md`, `docs/187-NAMES.md`, `docs/187-COMMANDS.md`, `docs/187-AUTOCOMPLETE.md`, README, CHANGELOG, install docs | Synchronize | **Not touched** | Explicitly M7.4 (release-surface sync), which depends on every module above actually existing first. Rewriting public docs to describe skills that aren't built yet would mislead readers. |

Leaving it saying "Charlotte modules" one commit after the ecosystem index says NATASHA would be the exact inequity this pass exists to prevent. Only this one section changes — not a full AGENTS.md sync. |

+## Decision
+
+Ship Phase 0 (audit/DAG/ADRs) and Phase 1 (identity/foundation) only, fully
+and verified. Every deferred item above is catalogued, not silently
+dropped — see `docs/audits/natasha-v3-baseline-audit.md` § Deferred, not
+forgotten, and `docs/plans/natasha-v3-master-sprint.md` for the phase order
+a future session should follow.
+
+## Consequences

+- The repository is **internally consistent** at every commit in this
+  pass: nothing links to or claims a skill, route, or config value that
+  doesn't exist.
+- A future session executing Phase 2+ has an accurate starting map instead
+  of discovering these sequencing conflicts mid-implementation.
